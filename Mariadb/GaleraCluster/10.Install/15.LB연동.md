

# 15. KT Cloud LB 정리



## **1. 개요**



KT Cloud의 Load Balancer는 일반적인 AWS/Azure의 LB와 달리

**Public IP + Port Forwarding 기반으로 동작하는 구조**를 가진다.



### **기본 트래픽 흐름**

```
Client → LB (Public IP) → Public IP:Port → Port Forwarding → VM Private IP:Port
```

즉,

- LB는 VM의 Private IP를 직접 대상으로 하지 않음
- **Public IP + Port 기반으로 Backend를 구성**





## **2. 동작 방식 핵심**

### **Backend 등록 조건**



KT Cloud LB는 다음 조건을 만족하는 서버만 등록 가능

- VM이 존재
- Public IP가 할당되어 있음
- 해당 VM에 **Port Forwarding 설정이 되어 있음**



👉 즉, 아래 조건 필수:

```
Public IP + Port Forwarding 설정된 서버만 LB 대상 가능
```



### **Backend 등록 단위**

- VM 단위가 아니라
- **Public IP + Port 단위**



예시:

```
1.1.1.1:20001 → VM1:80
1.1.1.1:20002 → VM2:80
```





## **3. AWS / Azure와 구조 차이**



| **구분**        | **KT Cloud**           | **AWS / Azure**     |
| --------------- | ---------------------- | ------------------- |
| 네트워크 모델   | NAT 기반               | VPC / VNet (SDN)    |
| Backend 대상    | Public IP + Port       | Private IP          |
| LB 역할         | Port Forwarding 분산   | 네트워크 레이어 LB  |
| 기본 보안 모델  | 사용자 책임            | 기본 차단 (Private) |
| Private LB 지원 | ❌ 제한적 / 사실상 없음 | ✅ 지원              |





## **4. 주요 한계점 (⚠️ 중요)**



## **4.1 Port Forwarding 기반 구조**



👉 가장 큰 특징이자 한계

- LB Backend로 등록하려면

  → 반드시 **Port Forwarding을 통해 외부 포트 개방 필요**



즉,

```
DB, WAS 등의 내부 서비스도
외부에서 접근 가능한 형태로 열어야 함
```





## **4.2 보안 취약성**

이 구조로 인해 다음과 같은 문제가 발생



### **❌ 의도치 않은 외부 노출**



- DB 포트 (ex. 3306)
- 내부 API 포트
- 관리 포트

👉 Public IP에 노출될 가능성 존재



### **❌ 공격 표면 증가**

- 포트 스캔 대상 증가
- 무차별 접근 시도 가능
- 방화벽 미설정 시 직접 공격 가능



### **❌ LB가 보안 경계 역할을 하지 못함**

AWS/Azure:

```
Internet → LB → Private VM (보호됨)
```

KT Cloud:

```
Internet → Public IP → VM (직접 노출됨)
           ↘ LB도 동일 경로 사용
```

👉 LB가 “보호 계층”이 아니라

👉 단순 “트래픽 분산 장치” 수준





## **4.3 Private 환경에서 사용 어려움**



KT Cloud LB는 구조적으로:

- Private IP 기반 Backend 연결 불가
- 내부망 전용 LB (Internal LB) 개념 부재



👉 결과:

```
AP → LB → DB (내부 구조)
```

❌ 구현 어려움





## **4.4 아키텍처 제약**



다음과 같은 구조 구현이 어려움:

- Internal Load Balancer
- Service Mesh 형태
- Zero Trust 구조
- DB 전용 내부 LB





## **5. 권장 설계 방식 (실무 대응)**



## **5.1 기본 원칙**

👉 **LB를 보안 장비로 간주하면 안됨**



## **5.2 권장 패턴**



### **✅ 1. 외부 서비스만 LB 사용**

- Web / API 서버만 LB 연결
- DB 등 내부 서비스는 제외





### **✅ 2. Port Forwarding 최소화**

- 꼭 필요한 포트만 개방
- 임의 포트 사용 (예: 20001 등)



### **✅ 3. 방화벽 필수 적용**

- 허용 IP 제한
- LB IP만 허용



### **✅ 4. 내부 통신은 Private IP 직접 사용**



```
AP → DB (Private IP)
```

👉 LB 사용하지 않음





### **✅ 5. 내부 LB 필요 시 우회**



#### **방법 1: Proxy VM 구성**

```
AP → HAProxy / ProxySQL → DB
```



#### **방법 2: DB 전용 솔루션**

- MariaDB MaxScale
- ProxySQL
- HAProxy





## **6. 최종 정리**



### **KT Cloud LB 특징**



- Public IP 기반 LB
- Port Forwarding 필수
- Private IP 직접 연결 불가





### **핵심 한계**

👉 **“Port Forwarding 기반 구조”**



이로 인해:



- 보안 취약성 증가
- 내부 서비스 노출 위험
- Private LB 구현 어려움
