# Load Balancer 





# **📌 1. 아키텍처 비교**







## **🔹 기존 (On-prem 스타일)**



```
Client → HAProxy → Galera(3node)
```



------





## **🔹 CSP 방식 (추천 ⭐)**



```
Client → KT Cloud Load Balancer → Galera(3node)
```



------





# **📌 2. 어떻게 동작하나?**



👉 LB가 하는 역할



- TCP 3306 트래픽 분산
- 장애 노드 자동 제외
- 단일 endpoint 제공





------

# **📌 4. 접속 방법**





👉 LB IP가 172.27.0.10이라면

```
mariadb -h 172.27.0.10 -u appuser -p
```

👉 클라이언트 입장에서는 그냥 DB 1개처럼 보임



------





# **📌 5. HAProxy vs LB 비교**



| **항목**    | **HAProxy** | **KT Cloud LB** |
| ----------- | ----------- | --------------- |
| 운영        | 직접 관리   | CSP 관리        |
| 장애 대응   | 직접 설정   | 자동            |
| 비용        | VM 필요     | LB 비용         |
| 설정 난이도 | 중간        | 쉬움            |

👉 결론: **CSP 환경에서는 LB가 더 좋음**





# **📌 6. ⚠️ 중요한 주의사항**







## **🔥 1. Galera는 Write 충돌 있음**





👉 LB로 분산하면:



- 3개 노드에 동시에 write 들어감
- 충돌 가능





------





## **🔥 2. 그래서 실무에서는**





👉 보통 이렇게 구성





### **방법 A (간단)**





- LB → 모든 노드
- PoC OK





------





### **방법 B (추천 ⭐)**





- Write용 LB → 1노드
- Read용 LB → 여러 노드





------
