# 1. Galera 백업 핵심 개념

## Galera 백업 시 주의사항

일반 MariaDB와 다르게 Galera Cluster에서는 다음을 고려해야 합니다.

| 항목 | 설명 |
|------|------|
| `--single-transaction` | InnoDB 일관성 스냅샷 (테이블 잠금 없이 백업) |
| `wsrep_desync=ON` | 백업 중 해당 노드가 cluster flow control 에 영향을 주지 않도록 분리 |
| 복원은 1개 노드만 | 1개 노드에 복원하면 Galera가 나머지 노드에 자동 동기화 |

## wsrep_desync 란?

```
[백업 중 without desync]         [백업 중 with desync]

Cluster → 노드A(백업중) 느림      Cluster → 노드A(desync) → 백업 독립 실행
         → 전체 cluster 영향              → 나머지 노드 영향 없음
```

백업 중에는 반드시 `wsrep_desync=ON` 후 백업, 완료 후 `OFF` 합니다.

# 2. 백업

## 사전 준비

```sh
cd Mariadb/GaleraCluster/30.백업-복원
chmod +x backup.sh restore.sh
```

## 백업 실행

```sh
# loadtest DB 백업 (기본값)
./backup.sh

# 다른 DB 백업
./backup.sh mydb
```

# 3. 복원

## 3-1. 복원 전 데이터 확인 (현재 상태)

```sh
docker exec mariadb-client \
  mariadb -u appuser -pAppPass1234! -h 14.63.223.198 -P 3306 \
  -e "SELECT COUNT(*) AS '현재 행 수' FROM loadtest.load_test;"
```

## 3-2. 테스트용: 데이터 삭제 후 복원

복원 테스트를 위해 데이터를 먼저 지웁니다.

```sh
docker exec mariadb-client \
  mariadb -u appuser -pAppPass1234! -h 14.63.223.198 -P 3306 \
  -e "DROP DATABASE loadtest;"
```

## 3-3. 복원 실행

```sh
./restore.sh backups/loadtest_20260407_120000.sql.gz loadtest
```

# 4. 복원 후 Galera 동기화 검증

복원은 LB를 통해 1개 노드에만 적용되지만, Galera가 나머지 노드에 자동 동기화합니다.

각 노드에 직접 접속해서 데이터가 동일한지 확인합니다.

```sh
# node1
docker exec mariadb-client \
  mariadb -u appuser -pAppPass1234! -h 172.27.0.24 -P 3306 \
  -e "SELECT COUNT(*) AS 'node1 행 수' FROM loadtest.load_test;"

# node2
docker exec mariadb-client \
  mariadb -u appuser -pAppPass1234! -h 172.27.0.53 -P 3306 \
  -e "SELECT COUNT(*) AS 'node2 행 수' FROM loadtest.load_test;"

# node3
docker exec mariadb-client \
  mariadb -u appuser -pAppPass1234! -h 172.27.0.175 -P 3306 \
  -e "SELECT COUNT(*) AS 'node3 행 수' FROM loadtest.load_test;"
```

3개 노드의 COUNT 결과가 동일하면 정상입니다.

백업 파일을 직접 확인하고 싶을 때:

```sh
# 앞부분만 확인
gunzip -c backups/loadtest_20260407_120000.sql.gz | head -50

gunzip -c backups/loadtest_20260407_120000.sql.gz | wc -l
```

# 6. Trouble Shooting

## wsrep_desync 권한 오류

`appuser` 계정은 `SUPER` 권한이 없어서 `wsrep_desync` 설정을 변경하지 못할 수 있습니다.

```sh
# node1 에서 root로 직접 실행
ssh root@172.27.0.24 "mariadb -uroot -e 'SET GLOBAL wsrep_desync=ON;'"
```

또는 `appuser`에 SUPER 권한 부여:

```sql
GRANT SUPER ON *.* TO 'appuser'@'%';
FLUSH PRIVILEGES;
```

## 복원 후 행 수가 0으로 보일 때

`table_rows` 는 InnoDB의 추정값입니다. 정확한 건수는 직접 확인합니다.

```sql
SELECT COUNT(*) FROM loadtest.load_test;
```

## Galera 동기화 지연 확인

복원 직후 다른 노드에서 바로 조회하면 아직 동기화 중일 수 있습니다.

```sql
-- 동기화 상태 확인
SHOW STATUS LIKE 'wsrep_local_recv_queue';
-- 0 이면 동기화 완료
```
