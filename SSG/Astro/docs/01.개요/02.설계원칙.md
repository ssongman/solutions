# 설계 원칙

Azure 랜딩존 설계는 Microsoft CAF가 정의한 **8가지 핵심 원칙**을 기반으로 합니다.  
모든 아키텍처 의사결정은 이 원칙에 따라 이루어져야 합니다.

---

## 8가지 설계 원칙

### 1. 구독 민주화 (Subscription Democratization)

구독은 **워크로드 격리와 확장의 단위**입니다.

- 팀/애플리케이션 단위로 구독 분리
- 중앙 팀이 구독 생성을 지연시키지 않아야 함
- 셀프서비스 구독 발급 프로세스 마련

```
❌ 나쁜 예: 모든 팀이 하나의 구독 공유
✅ 좋은 예: Dev / Staging / Prod 구독 분리
```

### 2. 정책 기반 거버넌스 (Policy-Driven Governance)

수동 검토 대신 **Azure Policy로 자동 적용**합니다.

- 리소스 배포 시 자동 정책 평가
- 비준수 리소스 자동 수정 (remediation)
- 자세한 내용: [정책 할당](../02.관리그룹/02.정책할당.md)

### 3. 단일 제어 및 관리 평면

모든 거버넌스는 **Azure Resource Manager(ARM)** 를 통해 일관성 있게 적용합니다.

### 4. 애플리케이션 중심 서비스 모델

플랫폼 팀은 **애플리케이션 팀이 빠르게 배포**할 수 있도록 기반을 제공합니다.

### 5. Azure 네이티브 설계

- PaaS 서비스 우선 활용
- 허브-스포크 네트워크 → [허브-스포크 토폴로지](../03.네트워킹/01.허브스포크.md) 참조

### 6. 초기부터 Azure 하이브리드 고려

온프레미스 연결(ExpressRoute / VPN)을 초기 설계에 반영합니다.

### 7. 최소 권한 접근 원칙

RBAC을 통해 **필요한 권한만 부여**합니다. → [ID 관리](../04.보안/01.ID관리.md) 참조

### 8. 변화 수용 설계

랜딩존은 처음부터 완벽하지 않아도 됩니다. **반복적으로 개선**하는 구조로 설계합니다.

---

## 설계 영역 (Design Areas)

| 영역 | 주요 의사결정 |
|------|-------------|
| Azure Billing | EA / MCA 계정 구조 |
| ID & 접근 관리 | Azure AD 테넌트, RBAC |
| 관리 그룹 | 계층 구조, 정책 상속 |
| 네트워크 토폴로지 | 허브-스포크 vs VWAN |
| 보안 | Defender, Sentinel |
| 관리 (운영) | Log Analytics, 백업 |
